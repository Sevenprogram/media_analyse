# 内容策略中心页面设计

- 日期：2026-05-23
- 页面：内容策略中心（`key_insights`）
- 目标：实现一个贴近参考图的企业中后台内容策略驾驶舱页面，替换当前占位页，并与现有 WebUI 壳层保持一致。

## 1. 背景与目标

当前系统已有较完整的中后台壳层，包括左侧导航、顶部全局头部，以及多个已落地的运营分析页面。`内容策略中心` 当前仍为占位态，无法承接“选题判断、内容结构复用、竞品样本参考、风险提示、流量预估”等核心内容策略工作。

本次设计目标：

- 用单页驾驶舱形式承载内容策略分析
- 风格尽量贴近用户提供的参考图，采用成熟企业中后台视觉语言
- 保持与现有项目的导航、头部、卡片体系兼容
- 先用高质量静态 mock 数据完成成品感展示
- 为后续接入真实接口保留清晰的数据结构边界

不在本次范围内的内容：

- 不接入后端接口
- 不实现真实图表库交互
- 不扩展为多页面流程产品
- 不改动全局导航、全局头部和其他页面的信息结构

## 2. 设计原则

### 2.1 视觉方向

页面采用“浅底白卡、青绿强调、信息密度较高”的企业中后台风格，尽量贴近参考图，而不做偏品牌展示型或实验性设计。

关键视觉原则：

- 页面背景保持浅灰白，顶部加入轻微高光和柔和渐变，避免界面发平
- 卡片统一使用白底、大圆角、细描边、轻阴影
- 主色使用青绿系，用于核心数字、正向趋势、激活态和主要操作
- 风险与下降使用橙/红进行语义表达
- 字号偏克制，依靠对齐、留白和色彩层次组织密集信息

### 2.2 产品感原则

- 一屏内展示“策略判断 + 证据支撑 + 可执行建议”
- 让每张卡片都具备明确角色，而不是纯装饰图块
- 优先强化“像成品”的完成度，再考虑复杂动态能力
- 延续现有系统交互习惯，避免新页面显得像另一个产品

## 3. 页面信息架构

页面作为 `key_insights` tab 对应主页面，在全局壳层内部渲染。

整体结构分为四个层次：

1. 页面标题区
2. 页面筛选区
3. 三段式驾驶舱内容区
4. 页面底部总结/导出动作区（如实现需要可简化为标题区动作）

### 3.1 页面标题区

用途：

- 明确当前页面为“内容策略中心”
- 展示页面更新时间和报告导出入口
- 营造参考图中的运营驾驶舱感

内容建议：

- 标题：内容策略中心
- 副信息：今日 09:30 更新 / 策略分析基于近 30 天样本
- 动作：导出报告、查看全部、刷新策略

### 3.2 页面筛选区

位于标题下方，形式为一排轻量筛选控件。

建议字段：

- 平台：全部平台 / 抖音 / 小红书 / 视频号 / B站
- 时间范围：近 7 天 / 近 30 天 / 近 90 天
- 内容目标：获客转化 / 种草互动 / 品牌声量
- 目标人群：宝妈（0-6 岁）/ 新手养宠 / 成分党等
- 内容阶段：冷启动 / 放量期 / 复盘优化
- 高级筛选：作为弱按钮入口

第一版行为：

- 仅维护前端本地状态
- 可触发页面内 mock 数据的轻量切换
- 不发起真实请求

## 4. 核心内容区设计

内容区按参考图组织为三段。

### 4.1 第一段：今日重点判断

#### 4.1.1 今日重点选题卡

位置：左上

用途：

- 用四个数字卡快速说明当前最值得关注的内容机会结构

建议指标：

- 高机会
- 高互动
- 低竞争
- 趋势上升

交互：

- hover 仅轻微抬升和描边增强
- 不做复杂点击态

#### 4.1.2 痛点分布环图卡

位置：中上

用途：

- 展示“宝妈/家长痛点分布”或其它人群议题分布

内容结构：

- 左侧或中间为环图
- 中心显示痛点总数或样本总量
- 右侧展示图例和占比

第一版实现方式：

- 使用 CSS conic-gradient 或 SVG 绘制
- 图例为静态 mock 数据

#### 4.1.3 热门关键词趋势榜

位置：右上

用途：

- 给出关键词、热度、趋势、机会值的排行榜视角

列建议：

- 排名
- 关键词
- 搜索热度
- 趋势
- 机会值

趋势展示：

- 用小型 SVG sparkline 表达上升或下降
- 上升使用绿色，下降使用红色

### 4.2 第二段：策略素材与建议

#### 4.2.1 内容框架库

位置：左中

用途：

- 展示可复用内容模板或主题切入方式

每行建议字段：

- 主题标题
- 参考标签
- 互动中位数
- 转化人数或线索量
- 操作按钮

操作可包含：

- 使用框架
- 查看更多

#### 4.2.2 选题建议

位置：中中

用途：

- 这是页面核心卡片，承接“基于热点、人群与竞品推导出的建议选题”

每行建议字段：

- 标题建议
- 目标人群
- 机会值
- 风险等级
- 操作

操作建议：

- 加入草稿
- 生成文案
- 查看证据

视觉重点：

- 机会值用高对比数字或色块强调
- 风险等级做弱色语义标签

#### 4.2.3 高表现同行内容样本

位置：右中

用途：

- 让运营在策略判断后，立即看到表现好的样本内容

每行建议字段：

- 平台图标
- 缩略图
- 标题
- 互动
- 点赞
- 评论
- 收藏

交互：

- 第一版仅展示静态列表和“查看全部”

### 4.3 第三段：风险、组合与流量预估

#### 4.3.1 风险提醒

位置：左下

用途：

- 提示策略执行中的主要风险点

每个风险项应包含：

- 风险标题
- 风险说明
- 风险等级
- 相关数量或影响值

视觉语义：

- 高风险：红
- 中风险：橙
- 一般提醒：黄或灰橙

#### 4.3.2 本周内容组合建议

位置：中下

用途：

- 用环图 + 表格形式展示本周建议的内容配比

内容建议：

- 痛点内容
- 科普内容
- 案例内容
- 转化内容

附加信息：

- 内容方向占比
- 建议篇数
- 预估互动量
- 预估线索量

#### 4.3.3 预期流量分布

位置：右下

用途：

- 预估不同平台可贡献的流量结构

内容形式：

- 环图
- 平台图例及占比
- 总预估曝光
- 预估转化订单或线索量

## 5. 视觉系统规范

### 5.1 色彩

页面使用语义化 token，避免在组件里散落裸色值。

建议色系：

- 主背景：浅灰白
- 卡片背景：纯白
- 主文字：深灰绿黑
- 次文字：中性灰绿
- 线框：低对比冷灰线
- 主强调：青绿
- 辅助强调：蓝绿、浅青
- 风险：橙、红

### 5.2 圆角与阴影

- 主卡片圆角偏大，参考 18px 到 24px
- 控件圆角中等，参考 12px 到 16px
- 阴影轻，不做悬浮夸张效果
- 描边始终存在，确保白卡在浅底上可识别

### 5.3 字体层级

- 页面标题：28px 到 32px，偏粗
- 卡片标题：15px 到 16px
- 主数字：28px 到 36px
- 表格内容：12px 到 13px
- 标签/辅助说明：11px 到 12px

### 5.4 图表策略

第一版图表目标是“视觉可信、风格统一、实现可控”。

建议：

- 环图：CSS conic-gradient 或 SVG
- 趋势线：小型 SVG path
- 占比条：纯 CSS 进度条
- 不引入重型图表配置以避免样式偏离参考图

## 6. 交互与状态

### 6.1 基础交互

- 筛选器切换有明确选中态
- 表格行 hover 有浅底反馈
- 卡片 hover 只做边框和阴影细微变化
- “查看更多”“查看全部”统一为轻量文本按钮或幽灵按钮

### 6.2 数据状态

第一版只做前端静态数据，但组件需要支持三种状态：

- 正常数据态
- 空态
- 长文本/高密度兜底态

### 6.3 空态要求

如果某卡片没有数据：

- 保留卡片框架
- 显示简短空态文案
- 不让页面布局塌陷

## 7. 响应式策略

### 7.1 桌面端

优先还原参考图观感，使用多列混排：

- 第一段三列
- 第二段三列
- 第三段三列

### 7.2 中等宽度

当宽度不足时，退化为两列：

- 右侧榜单和样本卡下移
- 环图和建议卡优先保留可读宽度

### 7.3 移动端

移动端改为单列流式布局：

- 卡片纵向堆叠
- 表格类卡片改为紧凑列表或横向可滚区域
- 筛选器换行

目标：

- 不出现横向破版
- 不让核心数字和标题挤压重叠

## 8. 技术实现设计

### 8.1 页面挂载方式

在 `main.tsx` 中将 `key_insights` tab 的占位页替换为真实页面组件。

建议新增：

- `api/webui/src/pages/ContentStrategyCenterPage.tsx`

### 8.2 组件拆分建议

页面内部可拆为以下局部组件或局部渲染函数：

- `StrategyHeader`
- `StrategyFilters`
- `StrategyMetricCards`
- `StrategyDonutCard`
- `StrategyKeywordTrendCard`
- `StrategyFrameworkLibraryCard`
- `StrategyTopicSuggestionCard`
- `StrategyCompetitorSampleCard`
- `StrategyRiskCard`
- `StrategyMixRecommendationCard`
- `StrategyTrafficDistributionCard`

第一版不强制拆成多个独立文件，可先在单文件中按模块组织；如果代码长度明显超过可维护范围，再提炼局部组件。

### 8.3 样式落点

样式放入现有 `api/webui/src/styles.css`。

命名要求：

- 使用独立前缀，避免和已有页面冲突
- 建议前缀：`ks-` 或 `cs-`

推荐本次使用：

- `ks-`（代表 key insights / strategy center）

### 8.4 数据模型

页面顶层维护 mock 数据常量与筛选状态。

建议包含的数据块：

- 筛选器选项
- 顶部指标卡数据
- 痛点分布数据
- 关键词趋势榜数据
- 内容框架库数据
- 选题建议数据
- 同行样本数据
- 风险提醒数据
- 本周内容组合数据
- 预期流量分布数据

后续接接口时，只需要把这些 mock 数据替换为接口返回结构。

## 9. 错误处理与可维护性

### 9.1 错误处理

因为第一版无真实请求，错误处理重点在界面稳态：

- 长标题截断或换行
- 表格容器可滚动
- 卡片数据为空时保留骨架
- 环图百分比容错到总和不严格等于 100 时仍可展示

### 9.2 可维护性

- 所有页面数据常量集中在页面文件顶部或单独 mock 模块
- 颜色、圆角、阴影尽量复用全局 token，新增时使用语义变量
- 避免把通用样式写成过宽选择器，防止污染其他页面

## 10. 验收标准

页面完成后应满足以下标准：

- `key_insights` 不再显示占位卡，而是完整策略驾驶舱
- 视觉效果明显贴近用户参考图
- 与现有侧边栏、全局头部、其他页面不割裂
- 桌面端具有成熟中后台的层级和密度
- 中等宽度与移动宽度下不破版
- 构建通过

## 11. 实施顺序

建议实施顺序如下：

1. 新建页面组件并接入 `main.tsx`
2. 先搭整体骨架和三区布局
3. 实现卡片与表格基础结构
4. 实现环图、趋势线、比例条等轻量图形
5. 打磨配色、间距、边框、阴影和 hover 反馈
6. 做响应式收敛和构建验证

## 12. 风险与取舍

### 风险

- 单文件页面内容较多，若无清晰分区容易失控
- 纯 CSS/SVG 图表若处理不好，会显得像占位图而不够成品
- 若直接照搬截图细节，可能与现有系统样式产生轻微割裂

### 取舍

- 优先保证整体观感像成熟产品，而不是追求 1:1 像素复刻
- 优先保证页面成品感，再考虑复杂动态筛选逻辑
- 优先保持与现有系统兼容，而不是完全另起一套设计语言

## 13. 结论

本次页面应采用“参考图骨架 + 现有系统融合”的方式实现：在保留企业中后台驾驶舱布局、信息密度和色彩语义的基础上，做成一个可直接替换现有占位页的高完成度页面。第一版以静态数据完成结构、视觉和交互基线，为后续真实策略接口接入预留清晰边界。
