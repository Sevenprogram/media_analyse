## 1. 总体原则

当前增长情报前端有两套数据口径同时存在：

1. **快照口径**
   - 优先读取 `keyword_heat_snapshots`
   - `content_tracking_snapshots`
   - `competitor_composition_snapshots`
   - `creator_daily_snapshots`

2. **原始帖子回退口径**
   - 当快照为空时，从 `research_posts` / `research_comments` 的真实采集数据推导图表和机会榜
   - 当前总览页、关键词热度页、报告页已支持这类回退

这意味着：

- 页面有图，不一定代表已经有正式快照表
- 页面为空，也不一定是前端坏了，可能是当前页仍依赖某类尚未生成的数据

## 3. 研究任务基础图表

研究任务页和部分总览图表会直接使用 `build_chart_summary()`。

代码位置：

- [D:/program/media_analyse/MediaCrawler/research/charts.py](D:/program/media_analyse/MediaCrawler/research/charts.py)

### 3.1 平台分布图

字段：

- 帖子数：按 `posts[].platform` 计数
- 评论数：按 `comments[].platform` 计数

公式：

```text
posts(platform) = count(posts where post.platform == platform)
comments(platform) = count(comments where comment.platform == platform)
```

输出结构：

```json
[
  { "platform": "xhs", "posts": 100, "comments": 20 },
  { "platform": "dy", "posts": 50, "comments": 5 }
]
```

### 3.2 帖子日趋势图

字段：

- `post.publish_time`

公式：

```text
daily_posts(date) = count(posts where publish_time.date == date)
```

限制：

- `publish_time` 不是 `datetime` 的记录会被忽略

### 3.3 评论日趋势图

字段：

- `comment.publish_time`

公式：

```text
daily_comments(date) = count(comments where publish_time.date == date)
```

### 3.4 关键词排行图

字段：

- `post.engagement_json.source_keyword`

公式：

```text
keyword_count(keyword) = count(posts where source_keyword == keyword)
```

说明：

- 这里不是对正文分词
- 是对采集阶段写入的 `source_keyword` 做计数
- 所以它更接近“采集入口关键词分布”，不是“内容文本热词”

### 3.5 AI 情绪 / 立场 / 主题分布

字段来源：

- `ai_results[].result_json.sentiment`
- `ai_results[].result_json.stance`
- `ai_results[].result_json.topic_tags`

公式：

```text
distribution(value) = count(ai result contains value)
```

限制：

- 如果没有 AI 分析结果，这三张图为空

## 4. 增长总览页

代码位置：

- [D:/program/media_analyse/MediaCrawler/research/dashboard.py](D:/program/media_analyse/MediaCrawler/research/dashboard.py)
- [D:/program/media_analyse/MediaCrawler/api/routers/reports.py](D:/program/media_analyse/MediaCrawler/api/routers/reports.py)

总览页不是单纯展示表，而是先把多种对象统一转成 `opportunity`，再排序。

### 4.1 监控卡片

#### 4.1.1 运行任务数

公式：

```text
running_jobs = count(jobs where status == "running")
```

#### 4.1.2 异常数

公式：

```text
errors = count(jobs where status in ["failed", "error"])
```

#### 4.1.3 今日采集量

当前口径：

```text
today_collected =
  sum(content_snapshots.total_content_count)
  + sum(competitor_compositions.total_flow_count)
```

说明：

- 这不是严格的“今日新增帖子数”
- `content_snapshots.total_content_count` 是内容追踪候选内容量
- `competitor_compositions.total_flow_count` 是友商快照里的总互动量
- 因此当前这个指标更接近“今日已汇总的内容规模和流量规模”

#### 4.1.4 最后更新时间

从以下对象中取最大时间：

- `keyword_heat_snapshots`
- `competitor_compositions`
- `content_snapshots`

### 4.2 今日增长判断

总览页会先构造机会列表，再生成摘要判断。

#### 4.2.1 样本不足判断

条件：

```text
if opportunities.length == 0:
    sample_status = "insufficient"
    confidence = "low"
```

输出文案：

- 当前样本不足
- 建议先执行一次实时发现

#### 4.2.2 样本充足判断

字段：

- `top = opportunities[0]`
- `evidence_count = sum(opportunity.evidence_count)`

公式：

```text
sample_status = "enough" if evidence_count >= 3 else "limited"
confidence = "high" if top.score >= 85 and evidence_count >= 6 else "medium"
```

headline 逻辑：

```text
"{platform}平台今日优先关注 {top.name}，{top.reason}"
```

### 4.3 综合机会榜

总览页把四类对象统一为机会：

1. 达人机会
2. 关键词机会
3. 友商机会
4. 内容机会

然后按 `score` 排序，取前 20。

## 5. 达人机会计算

### 5.1 分数公式

字段：

- `match_score`
- `recent_post_count_30d`
- `hot_post_rate`

公式：

```text
creator_score =
  match_score * 0.7
  + recent_post_count_30d * 1.0
  + hot_post_rate * 40

```

### 5.2 confidence

公式：

```text

medium if score >= 60 or evidence_count >= 1
```


## 6. 关键词机会与关键词热度

这里有两种相关但不同的口径：

1. **关键词机会榜**
2. **关键词热度信号**

### 6.1 关键词机会榜

来源有两种：

#### 6.1.1 标签体系正式口径

代码位置：

- [D:/program/media_analyse/MediaCrawler/research/competitors.py](D:/program/media_analyse/MediaCrawler/research/competitors.py)

输入：

- `tag_definitions`
- `entity_tags`
- `creator_profiles`
- `creator_daily_snapshots`

字段定义：

- `heat = tag_hit_counts + snapshot_distribution`
- `competition = creator_counts`
- `supply_gap = max(0, heat - competition)`
- `growth = snapshot_distribution`

公式：

```text
tag_hit_counts(tag) = count(entity_tags where tag_id == tag)
creator_counts(tag) = count(creator_profiles whose tag_summary_json contains tag)
snapshot_distribution(tag) = sum(snapshot.tag_distribution_json[tag])

heat_score = tag_hit_counts + snapshot_distribution
growth_score = snapshot_distribution
competition_score = creator_counts
supply_gap_score = max(0, heat_score - competition_score)
```

平台信号：

```text
boosting if heat >= 10 and growth >= 5
cooling  if heat <= 1 and growth <= 0
normal   otherwise
```

排序：

```text
sort by (supply_gap_score desc, heat_score desc)
```

#### 6.1.2 原始帖子回退口径

代码位置：

- [D:/program/media_analyse/MediaCrawler/api/routers/keyword_opportunities.py](D:/program/media_analyse/MediaCrawler/api/routers/keyword_opportunities.py)

当 `tag_definitions` 为空时：

1. 先遍历场景包关键词
2. 对每个关键词调用 `aggregate_keyword_heat_from_posts`
3. 如果场景包关键词也完全命不中，再退回到真实 `source_keyword` 排行

##### 场景词回退

每个场景词会生成：

- `heat_score = signal.heat_score`
- `growth_score = signal.medium_window.content_count`
- `competition_score = signal.medium_window.creator_count`
- `supply_gap_score = max(content_count - creator_count, 0)`

##### source_keyword 排行回退

当场景词结果为空时：

```text
source_keyword_count(keyword) =
  count(posts where engagement_json.source_keyword == keyword)
```

再对这些 `source_keyword` 计算热度信号，生成机会榜。

这是你当前页面里很多教育词能出图的关键原因，因为原始帖子很多不会直接写 “K12教育”，但会带 `source_keyword = 单亲妈妈陪读 / 新东方 / 家庭教育`。

### 6.2 关键词热度信号

代码位置：

- [D:/program/media_analyse/MediaCrawler/research/keyword_heat.py](D:/program/media_analyse/MediaCrawler/research/keyword_heat.py)

#### 6.2.1 匹配文本

判断一个帖子是否命中关键词时，使用：

```text
post_search_text =
  title + content + engagement_json.source_keyword + engagement_json.tag_list
```

#### 6.2.2 窗口聚合

对命中的帖子，计算三个窗口：

- 最近 24h
- 最近 7d
- 最近 30d

核心指标：

- `content_count`
- `engagement_total`
- `hot_post_count`
- `creator_count`

#### 6.2.3 ratio 计算

```text
content_ratio   = current_24h.content_count / avg_7d.content_count
engage_ratio    = current_24h.engagement_total / avg_7d.engagement_total
hot_ratio       = current_24h.hot_post_count / avg_7d.hot_post_count
creator_ratio   = current_24h.creator_count / avg_7d.creator_count
```

如果分母为 0：

- 当前值大于 0，则 ratio 记为 `2.0`
- 否则记为 `0.0`

#### 6.2.4 规则热度分

```text
heat_score =
  min(100, average(min(3, ratio_i)) / 3 * 100)
```

#### 6.2.5 推流分

```text
push_score =
  min(100,
    content_ratio * 22
    + engage_ratio * 28
    + hot_ratio * 30
    + creator_ratio * 20
  )
```

#### 6.2.6 限流 / 降温风险

```text
if content_ratio < 0.8 or engage_ratio < 0.8:
    cooldown_risk = max(0, 100 - push_score)
else:
    cooldown_risk = max(0, 35 - push_score / 4)
```

#### 6.2.7 标签判定

```text
limited  if cooldown_risk >= 65
cooling  if cooldown_risk >= 45
boosting if push_score >= 70
normal   otherwise
```

#### 6.2.8 样本置信度

根据：

- `content_24h`
- `content_7d`
- `creator_7d`

规则：

```text
high   if content_7d >= 100 and creator_7d >= 20 and content_24h >= 10
medium if content_7d >= 30 and creator_7d >= 8 and content_24h >= 3
low    otherwise
```

查询词会做简单拆分：

1. 统计关键词命中次数
2. 读取帖子标签命中
3. 两者之一有命中就保留

```text
keyword_component = min(60, sum(keyword_hit_count) * 20)
tag_component = min(40, tag_hit_count * 10)
coverage_component = matched_unique_terms / total_terms * 20

similarity_score = min(100, keyword_component + tag_component + coverage_component)
```

后端会调用总览页相同的原始帖子回退逻辑，从最近任务的帖子和评论里推导：

- 临时 `keyword_heat_snapshots`
- 临时 `content_snapshots`

这样报告页不会因为快照表还没生成而完全空白。

## 10. 总览页原始帖子回退逻辑

代码位置：

- [D:/program/media_analyse/MediaCrawler/api/routers/reports.py](D:/program/media_analyse/MediaCrawler/api/routers/reports.py)

函数：

- `_build_dashboard_fallback_from_jobs`

### 10.1 触发条件

当总览页请求时：

- `keyword_heat_snapshots` 为空
- 且 `content_snapshots` 为空

则回退。

### 10.2 回退过程

1. 读取最近任务
2. 对最近任务逐个拉取 `posts/comments`
3. 使用 `build_chart_summary()` 得到：
   - `keyword_ranking`
   - `post_trend`
   - `platform_counts`
4. 把 `keyword_ranking` 转成临时 `keyword_heat_snapshots`
5. 把帖子列表转成临时 `content_snapshots`

### 10.3 临时关键词热度快照

对每个来源关键词：

```text
heat_score = min(100, 35 + count * 4)
growth_score = min(100, count * 1.5)
platform_signal = "boosting" if count >= 10 else "normal"
```

这是**临时展示口径**，不是正式关键词热度模型。

### 10.4 临时内容快照

字段：

- `total_content_count = len(posts)`
- `hot_post_rate = estimate_hot_post_rate(posts)`
- `keyword_distribution = keyword_ranking top 12`

其中临时爆款率的阈值是：

```text
hot_post = engagement_total >= 100
```

## 11. 空图 / 空表的主要原因

当前页面可能为空，常见原因如下。

### 11.1 没有原始帖子

表现：

- 关键词页空
- 内容追踪页空
- 总览页样本不足

### 11.2 有原始帖子，但没有快照

如果该页面没做回退，就会空。

当前已做回退的页面：

- 总览
- 关键词热度
- 报告中心
- 内容追踪部分查询

当前仍较依赖配置或快照的页面：

- 友商监控
- 部分达人画像细分分析

### 11.3 关键词字面不命中

例如：

- 你输入的是 `K12教育`
- 真实帖子写的是 `英语启蒙 / 小升初 / 陪读 / 暑假班`

如果只做字面匹配就会空。当前内容追踪和关键词页已经部分通过 `source_keyword` 回退缓解了这个问题。

### 11.4 没有友商账号

友商页必须先有：

1. 友商账号
2. 友商帖子
3. 友商组成快照

否则图必然为空。

## 12. 当前限制与后续建议

### 12.1 当前限制

1. 同一系统里存在多套“热度”口径
   - 正式 `keyword_heat`
   - 原始帖子临时回退
   - source_keyword 排行回退

2. 今日采集量口径不纯
   - 混入了内容数和总互动量

3. 友商页后端已有更多分布字段，但前端还没全部画出来

4. 内容追踪页的 `matched_posts` 目前更偏“关键词命中”而不是“语义相似”

5. 报告摘要仍以规则模板为主，AI 只在单独的热点/选题链路参与

### 12.2 建议统一的后续方向

1. 统一“热度”口径
   - 报告、关键词页、总览全部优先同一套 `keyword_heat_snapshot`

2. 统一“爆款率”口径
   - 目前有固定 100 阈值
   - 也有 `max(100, avg * 2)` 阈值
   - 建议统一

3. 补齐友商图表
   - 关键词分布
   - 内容类型分布
   - 发布时间分布
   - 互动结构

4. 补齐达人画像图表
   - 平台分布
   - 粉丝分层
   - 活跃度分布
   - 爆款率分布

5. 对外展示时增加“口径说明”
   - 特别是老板页
   - 避免把临时回退结果误认为正式统计结果

## 13. 结论

当前系统已经能展示一批真实可用的图表，但这些图表背后不是单一的一套严格指标，而是：

- 快照优先
- 原始帖子回退兜底
- 规则分数与模板摘要混合

这套设计的优点是页面不容易空白，缺点是不同页面之间仍存在口径差异。后续如果要把这套系统作为老板汇报和运营决策主看板，建议下一阶段重点做“口径统一”和“友商 / 达人图表补齐”。
